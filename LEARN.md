# Altimeter 開發學習筆記：當設計美學遇上硬核架構

想像一下，你正在建造一台精密的瑞士手錶。手錶的外觀需要極致優雅（我們的 UI/UX），但內部的齒輪必須嚴絲合縫、互不干擾（我們的 Clean Architecture 乾淨架構）。這份文件，就是這支「數位手錶」的建造日誌與學習精華。

## 為什麼我們選這些工具？(技術決策)

在專案開始時，我們面臨了選擇：要用最原始的 HTML/JS 徒手打造，還是使用現代化的工具？

我們最終選擇了 **React + Vite**。
* **Vite**：就像是一台擁有 V8 引擎的超級跑車，它讓我們的開發與啟動速度快如閃電，不再需要像以前用 Webpack 時那樣泡杯咖啡等待。
* **React**：它讓我們可以把網頁拆成一個個「樂高積木」（Component）。如果我們要修改指南針的樣式，我們只需要去更換 `CompassCard` 這個積木，完全不會影響到其他部分。

## 內部結構：什麼是 Clean Architecture？

這聽起來很像某種潔癖，但它其實是軟體工程師保命的絕招。我們把專案拆成了三個「平行宇宙」：

1. **Domain (領域層 / `/src/domain`)**：
   這是合約中心。這裡定義了「我們需要什麼樣的數據？」（例如：需要經緯度、海拔、氣壓）。它完全不關心外面世界發生了什麼事，它只規定規則。
2. **Infrastructure (基礎設施層 / `/src/infrastructure`)**：
   這是黑手部門，專門和外部世界（瀏覽器 API、手機硬體）打交道。它把瀏覽器提供的各種感測器（如 `navigator.geolocation` 或 `Barometer`）封裝起來。
   * **巧思與退路 (Fallback)**：這也是我們處理「如果使用者的手機沒有氣壓計該怎麼辦？」的地方。我們寫了一個超酷的邏輯：如果手機說「抱歉我不支援氣壓計」，黑手部門就會馬上透過 GPS 座標，偷偷打一通電話給氣象局（Open-Meteo API），把當地的氣壓問回來。
3. **UseCase (應用邏輯層 / `/src/usecase`)**：
   這是交響樂團的指揮家。它負責協調「黑手部門」什麼時候該啟動、什麼時候該把數據傳給 UI。
4. **Presentation (UI / `/src/presentation` 與 `/src/App.tsx`)**：
   這是觀眾看到的舞台。我們用了 CSS 玻璃擬態（Glassmorphism）和深色模式，加上流暢的動畫，把指揮家送來的數據漂亮地展現出來。

## 最寶貴的經驗：我們學到了什麼？

### 1. 「優雅退路」是資深工程師的溫柔
**陷阱**：很多初階工程師在開發時，會假設「所有的功能都能完美運作」。結果到了 iOS 手機上，氣壓計直接崩潰。
**我們的解法**：在開發時，我們就設計了 `fetchPressureFromApi`。這就像是車子的備胎，我們可能永遠不想用到它，但當爆胎（硬體不支援）時，它可以讓旅程繼續。

### 2. 真實的物理感官 (UI/UX)
**亮點**：我們沒有直接印出死板的數字。我們用 CSS 的 `transform: rotate` 與 `transition`，幫指南針加上了「阻尼感」（就像真實的指南針一樣平滑過渡）。此外，我們讓背後的極光漸層網格，跟著手機的朝向產生微幅的「視差位移 (Parallax)」，這種微互動是區分「能用」跟「好用」的關鍵。

### 3. 先測試，再上戰場
我們寫了 **Vitest 單元測試** 來檢驗指揮家（UseCase）是否能在氣壓計故障時，精準地呼叫氣象局。這就像是在飛機起飛前的模擬機訓練，確保程式邏輯在面對極端狀況時不會出錯。

## 結語

開發這個 Altimeter 不只是一個簡單的程式作業，我們在過程中實踐了「關注點分離（Separation of Concerns）」、「防禦性編程（Defensive Programming）」以及「現代美學設計」。希望這支精緻的小程式，能帶給你對程式開發全新的想像！
